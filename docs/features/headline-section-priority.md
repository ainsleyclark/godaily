# Headline selection: section priority vs. score

This doc captures a known tradeoff in how the digest intro picks its
headline, and the upgrade path if it ever bites. Pick this up when the
intro starts dropping a genuinely hot story.

## Background

The digest subject + intro are generated by `Synthesise`
(`pkg/services/digest/prompts/title-intro.go`). The model only ever sees
the items that survive `filterItems`
(`pkg/services/digest/prompts/prompt.go`): at most 3 per source, capped
at 12 total. The headline rule in the prompt can therefore only choose
from whatever that filter sends.

Originally `filterItems` ordered and truncated the payload purely by
`Score`. Because `go_vuln` is the highest-weighted source
(`Source.Priority()` 23 in `pkg/domain/news/sources.go`), on a quiet day
its advisories scored near the top and crowded low-scored proposals out
of the payload entirely. The model then led with a routine security
advisory over a more significant proposal — not because it ignored the
rule, but because the proposal was never in the 12 items it received.

## Current behaviour (Option A)

`filterItems` now orders by **section priority first, then score within a
section**, and truncates on that order. Section priority comes from
`news.SectionTags` via the `sectionRank` helper — the same ordering the
headline prompt references, so the two cannot drift. Proposals (rank 1)
sit above Security (rank 9), so high-priority sections survive truncation
and lead the payload.

Pinned by the `Section Priority Beats Score` case in
`pkg/services/digest/prompts/prompt_test.go`.

## The tradeoff (why this might need revisiting)

Option A is a strict section ordering. On a **busy** day this can push a
very high-scoring discussion or article *out* of the 12-item payload in
favour of a low-scoring proposal that happens to sit in a higher-priority
section. In other words, we now guarantee section coverage at the
expense of occasionally dropping a genuinely hot lower-priority item.

This is an acceptable default — the headline should prefer releases and
proposals — but it is a blunt instrument.

## Upgrade path (Option B) if it bites

Guarantee the **top item of each high-priority section**, then fill the
remaining payload slots by raw score.

- **Approach:** in `filterItems`, first take the highest-scored item from
  each section in `news.SectionTags` order until a budget is reached
  (e.g. 1 per high-priority section), then fill the rest of `totalCap`
  from the remaining items sorted by score. This keeps a proposal/release
  in the running for the headline without evicting every hot discussion.
- **Tradeoff:** ~10 lines vs. Option A's sort comparator, and one more
  knob (how many guaranteed slots). Keep the `news.SectionTags` ordering
  as the single source of truth either way.
- **Tests:** extend `TestFilterItems` with a busy-day case asserting both
  a guaranteed proposal *and* the top discussions survive.

## Related

- Headline rule and `sectionRank`: `pkg/services/digest/prompts/`.
- Section ordering source of truth: `news.SectionTags` in
  `pkg/domain/news/item.go`.
- Source weighting (what made `go_vuln` win on score):
  `Source.Priority()` in `pkg/domain/news/sources.go`. Tuning weights
  there is an alternative lever, but it affects the whole digest
  ordering, not just the headline.
